## 一、字段命名规范

### 1.1 规则条件字段

预审规则的条件部分使用以下字段命名：

| 中文名称 | 英文字段名 | 数据类型 | 说明 | 示例值 |
|---------|-----------|---------|------|--------|
| 特征键 | featureKey | String | 特征的标识符，用于标识要判断的特征字段 | creditScore, orderAmount |
| 比较运算符 | compareOp | String | Compare Operator的缩写，用于比较运算 | >, <, >=, <=, =, != |
| 标签值 | tagValue | String | 特征的具体值或阈值 | 600, 10000 |

### 1.2 字段命名说明

#### featureKey（特征键）
- **命名来源**: feature + Key
- **含义**: 规则所判断的特征字段标识
- **示例**:
  - `creditScore` - 信用评分
  - `orderAmount` - 订单金额
  - `overdueDays` - 逾期天数
  - `accountAge` - 账龄

#### compareOp（比较运算符）
- **命名来源**: Compare Operator的缩写
- **放弃operator的原因**: operator过于通用，不够具体
- **采用compareOp的原因**: 更明确表示是比较运算，避免与其他操作符混淆
- **支持的运算符**:
  - `>` - 大于
  - `<` - 小于
  - `>=` - 大于等于
  - `<=` - 小于等于
  - `=` - 等于
  - `!=` - 不等于
  - `in` - 包含于
  - `notIn` - 不包含于

#### tagValue（标签值）
- **命名来源**: tag + Value
- **含义**: 特征的具体取值或阈值
- **示例**:
  - 数值型: `600`, `10000`
  - 字符串型: `"ACTIVE"`, `"SEAMLESS_V2"`
  - 数组型: `["A", "B", "C"]`

```json
{
  "featureKey": "creditScore",
```

**说明**: 只支持单个特征条件，不支持conditions数组和logic字段。

接口返回时将`rule_condition`展开为平铺字段：

```json
{
  "featureKey": "creditScore",
```

### 4.1 Java实体类

```java
@Data
public class RuleCondition {
    private String featureKey;

    // ✅ 只支持单个特征条件，不支持conditions数组和logic字段
```

### 4.3 字段转换逻辑

```java
/**
 * 将rule_condition JSON展开为平铺字段
 */
private void expandRuleCondition(PreAuditPolicyPo po, PreAuditPolicyVo vo) {
    try {
        RuleCondition condition = JSON.parseObject(
            po.getRuleCondition(),
            RuleCondition.class
        );

        vo.setFeatureKey(condition.getFeatureKey());
        vo.setCompareOp(condition.getCompareOp());
        vo.setTagValue(condition.getTagValue());
    } catch (Exception e) {
        // 复杂条件或不支持的条件类型
        vo.setFeatureKey(null);
        vo.setCompareOp(null);
        vo.setTagValue(null);
    }
}
```

### 5.3 规则匹配

```javascript
// 判断是否匹配规则
function matchRule(rule, userData) {
  const featureValue = userData[rule.featureKey];
  const compareOp = rule.compareOp;
  const tagValue = rule.tagValue;

  switch (compareOp) {
    case '>': return featureValue > tagValue;
    case '<': return featureValue < tagValue;
    case '>=': return featureValue >= tagValue;
    case '<=': return featureValue <= tagValue;
    case '=': return featureValue == tagValue;
    case '!=': return featureValue != tagValue;
    default: return false;
  }
}
```

## 六、命名规范对比

### 6.1 为什么不使用operator？

| 字段名 | 优点 | 缺点 | 结论 |
|--------|------|------|------|
| operator | 简短 | 过于通用，不够明确，容易与其他操作符混淆 | ❌ 不推荐 |
| compareOp | 明确表示是比较运算 | 稍长 | ✅ 推荐 |
| comparisonOperator | 最明确 | 过长，不够简洁 | ⚠️ 可用但不推荐 |

### 6.2 字段命名一致性

所有规则相关字段遵循统一命名规范：

| 功能分类 | 字段名 | 命名方式 |
|---------|--------|---------|
| 规则标识 | ruleId, ruleName, ruleType | 驼峰命名 |
| 规则条件 | featureKey, compareOp, tagValue | 驼峰命名 |
| 规则状态 | status, priority, description | 驼峰命名 |
| 时间戳 | createdAt, updatedAt | 驼峰命名 |

## 七、扩展性考虑

### 7.1 当前支持的规则类型

仅支持**特征规则**（Feature Rule），即基于用户特征进行比较判断。

### 7.2 多条件场景的处理方式

如果需要实现"信用评分>600且订单金额<10000"这种多条件判断，可以通过以下方式：

**方案1：创建两条规则（推荐）**

```json
// 规则1：信用评分>600
{
  "ruleId": "RULE_001",
  "ruleName": "信用评分>600",
  "priority": 11,
  "featureKey": "creditScore",
  "compareOp": ">",
  "tagValue": "600"
}

// 规则2：订单金额<10000
{
  "ruleId": "RULE_002",
  "ruleName": "订单金额<10000",
  "priority": 12,
  "featureKey": "orderAmount",
  "compareOp": "<",
  "tagValue": "10000"
}
```

**匹配逻辑**: 按优先级依次匹配，先匹配规则1，再匹配规则2。

**方案2：创建组合特征**

```json
// 在数据层面预先计算组合特征
{
  "ruleId": "RULE_003",
  "ruleName": "优质客户",
  "priority": 10,
  "featureKey": "isHighQualityCustomer",  // 组合特征标识
  "compareOp": "=",
  "tagValue": "true"
}
```

**说明**: 通过特征工程预先计算组合特征，在规则层面简化为单条件判断。

### 7.3 扩展性建议

1. **保持字段命名一致性**: 新增字段遵循驼峰命名规范
2. **单条件原则**: 每个规则只支持单个特征条件，保持简单性
3. **多条件处理**: 通过创建多条规则或特征工程实现多条件判断
4. **版本控制**: 接口返回数据包含版本号，便于兼容性处理
5. **文档同步更新**: 扩展功能时同步更新本文档

## 八、常见问题

### Q1: 为什么使用compareOp而不是operator？

**A**: operator过于通用，可能指代多种操作符（如算术运算符、逻辑运算符、位运算符等）。compareOp更明确地表示这是**比较运算符**，避免歧义。

### Q2: tagValue为什么不用value？

**A**: value同样过于通用，不够明确。tagValue表明这是特征的**标签值**或**阈值**，语义更清晰。

### Q3: 如何实现多条件判断（如AND、OR逻辑）？

**A**: 当前设计每个规则只支持单个特征条件。如需实现多条件判断，有两种方案：

1. **创建多条规则**: 按"且"逻辑创建多条规则，按优先级依次匹配
2. **特征工程**: 在数据层面预先计算组合特征，规则层面使用单条件判断

例如：实现"信用评分>600且订单金额<10000"
- 方案1: 创建两条规则（priority 11和12），按优先级匹配
- 方案2: 预先计算isHighQualityCustomer特征，规则判断该特征是否为true

### Q4: 数据库存储为什么用JSON？

**A**:
- ✅ 灵活性强，易于扩展
- ✅ 支持复杂条件（AND、OR等）
- ✅ 便于查询和解析
- ⚠️ 需要在接口层展开为平铺字段，提升可读性
