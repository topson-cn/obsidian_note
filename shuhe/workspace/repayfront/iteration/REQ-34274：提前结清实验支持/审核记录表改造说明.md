# 审核记录表改造说明

> 文档版本: v1.0
> 创建时间: 2026-03-02
> 说明: 提前结清审核模块审核记录表改造设计

---

## 一、需求背景

### 1.1 现状分析

**当前audit_record表**:
- ✅ 存储审核记录基本信息（用户ID、订单ID、审核状态等）
- ❌ 缺少flowId字段，无法实现全流程追踪
- ❌ 缺少match_rule_id字段，无法记录匹配的审核规则
- ❌ 无法支持AB实验数据追踪

### 1.2 需求目标

根据PRD需求，需要：
1. ✅ 在audit_record表新增flow_id字段
2. ✅ 在audit_record表新增match_rule_id字段
3. ✅ 实现预审→审核→还款全流程数据关联
4. ✅ 支持AB实验数据追踪和分析

## 二、数据库设计

### 2.1 审核记录表（audit_record）

**表名**: `repayfront.audit_record`

**改动类型**: 字段新增

**新增字段**:
```sql
ALTER TABLE repayfront.audit_record
ADD COLUMN flow_id VARCHAR(50) NOT NULL COMMENT '业务流水号（唯一标识一次提前结清申请全流程）',
ADD COLUMN match_rule_id VARCHAR(50) NULL COMMENT '匹配的审核规则ID（来自audit_policy表）';
```

**新增索引**:
```sql
-- flow_id唯一索引（一个flow_id对应一条审核记录）
CREATE UNIQUE INDEX uk_flow_id ON repayfront.audit_record(flow_id);

-- match_rule_id普通索引（用于按规则ID查询审核记录）
CREATE INDEX idx_match_rule_id ON repayfront.audit_record(match_rule_id);
```

**改动点**:
- ✅ flow_id字段关联预审记录表（pre_audit_record）和审核记录表（audit_record）
- ✅ match_rule_id字段记录匹配的审核规则ID，便于分析规则效果
- ✅ 支持AB实验数据追踪（通过flow_id关联全流程数据）

### 2.2 字段说明

| 字段名 | 类型 | 说明 | 示例值 |
|--------|------|------|--------|
| flow_id | VARCHAR(50) | 业务流水号（唯一标识一次提前结清申请） | TQJQ_20260302120000_abc123 |
| match_rule_id | VARCHAR(50) | 匹配的审核规则ID | RULE_001, RULE_002等 |

### 2.3 表结构完整示例

```sql
CREATE TABLE repayfront.audit_record (
    id BIGINT(20) UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '主键',
    flow_id VARCHAR(50) NOT NULL COMMENT '业务流水号（✅新增）',
    user_id VARCHAR(50) NOT NULL COMMENT '用户ID',
    order_id VARCHAR(50) NOT NULL COMMENT '订单ID（还款元素号）',
    audit_status VARCHAR(20) NOT NULL COMMENT '审核状态：PENDING-待审核，ACCEPTED-审核通过，REJECTED-审核拒绝',
    audit_rule_id VARCHAR(50) NULL COMMENT '审核规则ID（✅新增，记录匹配的审核规则）',
    auditor VARCHAR(50) NULL COMMENT '审核人',
    audit_time TIMESTAMP NULL COMMENT '审核时间',
    audit_finish_time TIMESTAMP NULL COMMENT '审核完成时间',
    audit_comment VARCHAR(500) NULL COMMENT '审核意见',
    created_by VARCHAR(150) NOT NULL DEFAULT 'SYS' COMMENT '创建人',
    updated_by VARCHAR(150) NOT NULL DEFAULT 'SYS' COMMENT '更新人',
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_flow_id (flow_id),  -- ✅ 新增
    INDEX idx_user_order (user_id, order_id),
    INDEX idx_audit_status (audit_status),
    INDEX idx_match_rule_id (match_rule_id),  -- ✅ 新增
    INDEX idx_created_at (created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='审核记录表';
```

### 3.3 Service层改造

**RepayAuditServiceImpl.java** - 审核提交逻辑改造：
```java
@Override
public RepayAuditResult repayAuditSubmit(String uid, String flowId, String repayElementNo,
                                         AuditAction auditAction, String auditComment) {

    // 1. 查询审核规则（现有逻辑）
    AuditPolicyPo policy = auditPolicyService.matchPolicy(uid, repayElementNo);

    // 2. 构建审核记录
    TAuditRecord record = TAuditRecord.builder()
        .flowId(flowId)                      // ✅ 设置flowId
        .userId(uid)
        .orderId(repayElementNo)
        .auditStatus(AuditStatus.PENDING)
        .matchRuleId(policy != null ? policy.getRuleId() : null)  // ✅ 设置匹配的规则ID
        .auditor(getCurrentUserId())
        .auditTime(new Date())
        .build();

    // 3. 根据审核动作更新状态
    if (auditAction == AuditAction.ACCEPT) {
        record.setAuditStatus(AuditStatus.ACCEPTED);
        record.setAuditFinishTime(new Date());
    } else if (auditAction == AuditAction.REJECT) {
        record.setAuditStatus(AuditStatus.REJECTED);
        record.setAuditFinishTime(new Date());
    }
    record.setAuditComment(auditComment);

    // 4. 保存审核记录
    auditRecordRepository.insert(record);

    return buildResult(record);
}
```

**核心改动点**:
1. ✅ 新增flowId参数，传递到Service层
2. ✅ 审核记录中保存flowId字段
3. ✅ 审核记录中保存matchRuleId字段（从匹配的审核规则中获取）
4. ✅ 实现预审→审核全流程数据关联

## 四、数据流转图

```

    ↓
前端生成flowId
    ↓
前端：调用/repayAudit/preSubmit接口（✅传递flowId）
    ↓
后端：预审判断
    ↓
后端：✅ 保存预审记录到pre_audit_record表（含flowId）
    ↓
    ├─ 无需审核 → 挽留判断 → 还款支付
    └─ 需要审核 → 用户确认提交审核
                 ↓
                 前端：调用/repayAudit/submit接口（✅传递flowId）
                 ↓
                 后端：审核判断
                 ↓
                 后端：✅ 保存审核记录到audit_record表（含flowId + match_rule_id）
                 ↓
                 ├─ 审核通过 → 挽留判断 → 还款支付
                 └─ 审核拒绝 → 结束
```
