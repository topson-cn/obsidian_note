# PH170130 - 筛选入账单

## 节点定位

**核心职责**：作为入账流程的"路由控制器",根据待入账单状态决定流程走向。

**设计模式**：流程控制节点(通过 initFacts 设置流程变量控制分支)

**处理策略**：
- 有待入账单 → 设置当前入账单,继续入账流程
- 无待入账单 → 标记完成,跳过入账流程

## 业务上下文

### 在流程中的位置

```
[PH170120 生成入账单] → [PH170130 筛选入账单] → 条件判断(fundIncomeFinished)
                                              ├─ false: 执行入账子流程
                                              │   ├─ PH170131 通知资方入账
                                              │   ├─ PH170132 获取资方入账明细
                                              │   ├─ PH170036V1 客账入账
                                              │   └─ PH170037 获取入账结果
                                              │       ↓
                                              │   [循环回 PH170130]
                                              └─ true: 跳过入账,进入后置处理
```

### 核心业务逻辑

#### 1. 入账单筛选规则

**数据来源**：`repayment_income_bill` 表

**筛选条件**：
- 还款单编号 = 当前还款单编号
- 入账单状态 = INIT(初始化)

**排序规则**：按 `income_seq_no`(入账顺序号) 升序

**选择策略**：取排序后的第一个入账单

#### 2. 流程控制逻辑

**分支判断**：通过设置 `fundIncomeFinished` 控制后续流程

| 场景 | fundIncomeFinished | currentIncomeBillNo | 流程走向 |
|------|-------------------|---------------------|---------|
| 有待入账单 | false | 第一个入账单编号 | 执行入账子流程 |
| 无待入账单 | true | null | 跳过入账流程 |

#### 3. 循环入账机制

```
第一次执行 PH170130:
  待入账单: [IB001(seqNo=1), IB002(seqNo=2)]
  → fundIncomeFinished=false
  → currentIncomeBillNo=IB001
  → 执行入账流程(IB001)

第二次执行 PH170130:
  待入账单: [IB002(seqNo=2)]  // IB001 已入账,状态变更
  → fundIncomeFinished=false
  → currentIncomeBillNo=IB002
  → 执行入账流程(IB002)

第三次执行 PH170130:
  待入账单: []  // 所有入账单已处理
  → fundIncomeFinished=true
  → currentIncomeBillNo=null
  → 跳过入账,流程结束
```

## 技术实现

### 核心方法：initFacts

**框架钩子**：在节点执行前自动调用,用于初始化流程变量

**执行时机**：
1. 每次进入节点时调用
2. 在 `process()` 方法之前执行
3. 用于设置流程分支的条件变量

**关键逻辑**：

```
输入: RepayApplyContext.currentRepaymentBillNo

Step1: 查询待入账单
  - 调用 repaymentIncomeBillService.getByRepaymentBillNo()
  - 条件: repaymentBillNo + incomeBillStatus=INIT

Step2: 判断是否有待入账单
  ├─ 有待入账单
  │   ├─ 按 incomeSeqNo 排序
  │   ├─ 设置 fundIncomeFinished = false
  │   └─ 设置 currentIncomeBillNo = 第一个入账单编号
  └─ 无待入账单
      ├─ 设置 fundIncomeFinished = true
      └─ 设置 currentIncomeBillNo = null

输出:
  - ProcessContext.funds.fundIncomeFinished
  - RepayApplyBo.currentIncomeBillNo
```

### 数据模型

#### RepaymentIncomeBill 核心字段

| 字段 | 类型 | 说明 | 业务含义 |
|------|------|------|---------|
| incomeBillNo | String | 入账单编号 | 唯一标识 |
| repaymentBillNo | String | 还款单编号 | 关联还款单 |
| incomeSeqNo | Integer | 入账顺序号 | 控制入账顺序 |
| incomeStatus | IncomeBillStatusEnum | 入账单状态 | INIT/PROCESSING/SUCCESS/FAILED |
| incomeType | IncomeBillTypeEnum | 入账类型 | 区分不同入账场景 |
| assetBank | BankEnum | 资方银行 | 资方标识 |
| assetId | String | 资产ID | 关联资产 |
| incomeOrderList | List | 入账订单列表 | 入账明细 |
| fundIncomeDetail | String | 资方入账明细 | JSON格式 |
| extInfo | IncomeBillExtInfo | 扩展信息 | 额外业务数据 |

#### IncomeBillStatusEnum 状态流转

```
INIT(初始化) → PROCESSING(入账中) → SUCCESS(入账成功)
                                  → FAILED(入账失败)
```

**PH170130 关注的状态**：仅查询 INIT 状态的入账单

## 关键设计点

### 1. 为什么需要筛选入账单?

**业务需求**：
- 一个还款单可能生成多个入账单(如:部分入账、分批入账)
- 需要按顺序依次处理每个入账单
- 支持循环入账,直到所有入账单处理完成

**技术实现**：
- 通过状态过滤(INIT)确保只处理待入账单
- 通过顺序号(incomeSeqNo)确保入账顺序
- 通过循环调用实现批量入账

### 2. 为什么核心逻辑在 initFacts 而不是 process?

**框架设计**：
- `initFacts()` 用于设置流程控制变量(Facts)
- `process()` 用于执行业务逻辑
- PH170130 的职责是"判断路由",不是"执行业务"

**最佳实践**：
- 流程控制节点: 逻辑在 initFacts, process 仅返回成功
- 业务处理节点: 逻辑在 process, initFacts 仅做初始化

### 3. 如何保证幂等性?

**幂等机制**：
- 每次执行都重新查询最新状态
- 只查询 INIT 状态的入账单
- 已处理的入账单状态已变更,不会被重复查询

**并发安全**：
- 入账单状态变更(PH170037)会更新为 SUCCESS/FAILED
- 下次查询时已处理入账单不会出现在结果中

### 4. 循环入账如何避免死循环?

**退出条件**：
- 无待入账单时设置 fundIncomeFinished=true
- 流程引擎根据此变量跳过入账子流程
- 进入后续节点,循环结束

**业务保障**：
- 每次入账都会更新入账单状态
- 状态为 INIT 的入账单数量有限
- 最多循环次数 = 入账单数量

## 性能与监控

### 数据库查询

**查询SQL**：
```sql
SELECT *
FROM repayment_income_bill
WHERE repayment_bill_no = ?
  AND income_bill_status = 'INIT'
ORDER BY income_seq_no ASC
```

**索引要求**：
```sql
KEY idx_repayment_bill_status (repayment_bill_no, income_bill_status)
```

**性能关注点**：
- 单次查询,无复杂关联
- 使用复合索引,查询效率高
- 入账单数量通常较少(1-5个)

### 监控指标

| 指标 | 含义 | 异常场景 |
|------|------|---------|
| 待入账单数量 | 每次查询返回的数量 | 数量过多(>10)需关注 |
| 循环次数 | 该节点在同一还款单中被调用的次数 | 次数过多(>10)可能死循环 |
| 查询耗时 | 数据库查询时间 | >100ms 需优化索引 |

### 日志记录

**关键日志**：
```
- 查询待入账单: repaymentBillNo, count
- 无待入账单: repaymentBillNo
- 设置当前入账单: incomeBillNo
```

## 异常处理

### 常见异常场景

| 异常 | 影响 | 处理方式 |
|------|------|---------|
| 数据库查询超时 | 无法判断是否有待入账单 | 框架捕获,流程失败 |
| 入账单数据不完整 | 后续节点可能失败 | 在 PH170131 等节点校验 |
| 状态数据不一致 | 查询结果与预期不符 | 在 PH170037 校验入账结果 |

### 异常恢复

**框架机制**：
- 流程失败后可重试
- 重试时重新执行 initFacts,重新查询
- 状态已变更的入账单不会重复处理

## 业务场景示例

### 场景1: 单笔入账

**输入**：
- 还款单: RP202401010001
- 入账单: [IB001(seqNo=1, INIT)]

**第一次执行**：
- 查询: [IB001]
- 设置: fundIncomeFinished=false, currentIncomeBillNo=IB001
- 流程: 执行入账子流程

**第二次执行**：
- 查询: [] (IB001 已入账成功)
- 设置: fundIncomeFinished=true, currentIncomeBillNo=null
- 流程: 跳过入账,流程结束

### 场景2: 多笔入账

**输入**：
- 还款单: RP202401010002
- 入账单:
  - IB001(seqNo=1, INIT, 金额=5000)
  - IB002(seqNo=2, INIT, 金额=3000)

**循环执行**：

| 执行次数 | 待入账单 | currentIncomeBillNo | fundIncomeFinished | 流程走向 |
|---------|---------|---------------------|-------------------|---------|
| 1 | [IB001, IB002] | IB001 | false | 入账IB001 |
| 2 | [IB002] | IB002 | false | 入账IB002 |
| 3 | [] | null | true | 流程结束 |

### 场景3: 无入账单

**输入**：
- 还款单: RP202401010003
- 入账单: []

**执行**：
- 查询: []
- 设置: fundIncomeFinished=true, currentIncomeBillNo=null
- 流程: 直接跳过入账,进入后置处理

## 最佳实践

### 开发注意事项

1. **不要修改状态**：该节点仅查询和判断,不修改入账单状态
2. **不要假设顺序**：必须按 incomeSeqNo 排序,不能依赖数据库默认排序
3. **处理空值**：currentIncomeBillNo 可能为 null,下游节点需判空
4. **幂等设计**：每次执行都重新查询,不依赖缓存

### 扩展建议

1. **监控告警**：循环次数 > 10 次告警,防止死循环
2. **性能优化**：入账单数量 > 5 时考虑批量处理
3. **数据一致性**：定期校验 INIT 状态入账单,防止状态卡死
4. **日志增强**：记录每次循环的入账单详情,便于排查问题
